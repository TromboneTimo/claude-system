NEVER stack multiple instances of the same causal pattern on a single slide. One instance + the universal principle beats three instances.

**The exact failure pattern (2026-04-13):**
After Timo corrected the triptych → roadmap layout, I over-rotated. Built slide 11 with THREE roadmap rows (Musician / Artist / Professor) + 4 stage columns + 12 data cells + footer. Timo: "visuals are also still way too complex on the roadmap. WHY the fuck is there so much information. Remove the 3 roadmaps and just have one roadmap and one single phrase so it's simple."

The right fix was ONE horizontal roadmap showing the universal 4-stage flow (Input → Signal → Action → Outcome) + one phrase below ("None of those videos were the biggest on their channels. They were the rightest.") That's the slide's argument in its minimum viable form.

**Why I kept over-stacking:**
1. When I move from one bad layout to a better layout, I over-weight "more proof = stronger." 3 examples FEEL more convincing than 1 principle. They're not. They just CROWD the slide.
2. I confuse "show the universal pattern" with "show every instance of the pattern." The universal pattern is the PRINCIPLE, not the roster of instances.
3. I conserve the old triptych's content when migrating to a new layout. The triptych had 3 items, so the roadmap got 3 rows. The items weren't the point — the mechanism was.

**The decision rule:**
Before adding a second instance of anything to a slide, ask: "does the second instance teach a new thing, or does it just repeat the first?" If it repeats, CUT IT. One clear arrow on one clear path beats three identical arrows stacked.

**Minimum viable slide test:**
- Could this slide communicate its argument with ONE visual element + ONE phrase? If yes, that's the target. Prove you need more before adding.
- Everything else is backing evidence that goes in a LATER slide or gets left out entirely.
- "Three examples" usually means three slides, each with one example, or one slide with one example + a phrase that generalizes.

**How to apply:**
- Before shipping any multi-instance slide, strip it to the MINIMUM: can I show the mechanism with one instance? If yes, that's the slide.
- For pattern-demonstration slides: "The same path, three different creators" is 3x harder to read than "The same path. Every time." with ONE clean visual of the path.
- If the user has 3 proof examples, put them on 3 separate slides OR leave them off the roadmap slide entirely. The roadmap slide teaches the PATTERN. Proof comes before or after.

**Why I missed this the first time:**
I was running visual QA against "does it render cleanly" and "does the content fit." I wasn't asking "is this the MINIMUM that makes the argument?" Minimum is a separate criterion and must be checked explicitly before shipping.
