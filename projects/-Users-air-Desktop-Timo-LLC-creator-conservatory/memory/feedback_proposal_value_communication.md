# Proposal Value Communication. Outcome-First Framing

Every proposal must communicate the VALUE the client gets, not the FEATURES being built. Seven rules, all of which were violated in the 2026-05-10 Victor Alegria draft and required a full rewrite.

## Rule 1. Lead with outcomes, not systems.

Every component description must be framed by:
- **What's broken right now** (specific to this client's situation, with a concrete hook)
- **What the client walks away with** (the outcome, in plain client-facing language)
- **Why this matters** (the strategic rationale tied to the engagement goal)

NEVER lead with "What Timo Builds" alone. The "What Timo Builds" line is the THIRD beat, not the first. Reader's first impression of every section is the pain it relieves and the win it produces.

**Why:** A proposal that opens "we're building a voice-of-customer database with a persona extraction layer" reads like an engineering spec. The buyer doesn't speak that language. They speak in problems they have and outcomes they want.

**How to apply:** Every system block in Section 2 follows this template, in order:
1. `**What's broken right now.**` Plain English. Names the client's actual situation, not a generic version.
2. `**What [client name] walks away with.**` Bullet list of outcomes the client can describe to a friend.
3. `**Why this matters.**` One paragraph tying the system to the engagement goal.

The previous template (`Problem` / `What Timo Builds` / `System Diagram` / `ROI`) is deprecated for client-facing copy because it leads with build mechanics.

## Rule 2. Find the specific pain hook before drafting.

Generic problem framing ("the data is fragmented across platforms," "every video starts with a blank page") reads as filler. The proposal needs a specific, evidence-backed hook unique to this client.

For Victor: ManyChat list sitting cold. Not "you have audience signals scattered across platforms." THE HOOK was "you have a list of opted-in subscribers and they're not being emailed; that's revenue sitting on the table every day."

**Why:** The buyer reads the proposal looking for "yes, that's me." A specific hook makes them say yes. A generic hook makes them skim.

**How to apply:** Before drafting Section 1, ask the user for ONE concrete pain hook per system. Write it down. Make sure it's verifiable (not invented). If the user can't name a specific hook for a given system, that system probably shouldn't be in the proposal.

## Rule 3. Consulting / advisory / ongoing access goes BEFORE pricing, always.

If the engagement includes ANY ongoing access (consulting, advisory, partnership, lifetime support, retainer-flavored access), it gets its OWN top-level section BEFORE the pricing section. Never after. Never bundled inside the pricing section as a sub-block.

**Why:** Section ordering controls what the reader anchors on. Consulting access is part of the value pile. The pricing section reads as "the cost of everything above." Burying consulting after pricing reads as an afterthought and weakens the value framing.

**How to apply:** Section structure for any engagement with ongoing access:
1. What Was Requested
2. What [Client] Gets (the systems / deliverables)
3. **[Ongoing Access Name]** ← own section
4. Engagement Structure & Pricing
5. Delivery Timeline
6. Protections & Terms
7. Next Steps

If there is no ongoing access, drop section 3 and renumber. Default scope where access is part of the deal: section 3 stays.

This is a hard structural rule, enforced regardless of the canonical reference (Harrison, Robinson's, Artifact). 2026-05-10 incident: Timo had to repeat this multiple times before the rule landed.

## Rule 4. Don't price-anchor the ongoing access.

Do NOT write "$X to $Y per month is the market rate for this kind of consulting." Do NOT calculate annualized value. Do NOT compare the consulting access to other consultants' retainers.

**Why:** Price-anchoring turns a relationship into a transaction. The buyer reads "$2K-$5K/month" and starts mentally subtracting that from the engagement total to compute what the build is "really" worth. That kills the deal psychology. Worse, it commits the seller to the implication that this access is meterable, which it isn't.

**How to apply:** When the consulting access is included, describe what's included, then state plainly: "Included. No expiration. No metered hours. No separate retainer." That's the framing.

The pricing rationale section CAN reference market comparisons for the BUILD work (e.g., "comparable infrastructure builds run $7,500 and up"). But the ongoing access never gets a sticker price.

## Rule 5. Order systems by client value, not by technical dependency.

Even if System B depends on System A internally, the proposal orders systems by which one delivers the client most value first. Internal dependencies are a build concern, not a sales concern.

**Why:** The reader spends most attention on the first system block. Put the highest-impact system there. For Victor, Email Engine leads because that's the unactivated-revenue hook. The fact that the email engine pulls from the content database internally is irrelevant to ordering.

**How to apply:** Before locking system order, ask: "If the client only read the first system block and stopped, which one would maximize their belief that this engagement is worth the price?" That system goes first.

## Rule 6. Surface goal-driven help as named deliverables.

If the engagement includes high-ticket-offer design, program structure consulting, strategy work, or any other premium intellectual deliverable, NAME IT as a top-line item, not a footnote inside "advisory access."

**Why:** Buyers undervalue what isn't named. Calling something "advisory access" telegraphs "I'll answer questions when you ask." Calling it "high-ticket offer design" telegraphs "I'm doing the work that turns your offer into something worth charging premium for." Same scope, different perceived value.

**How to apply:** In the consulting / ongoing access section (Section 3), list the named deliverables FIRST, before the always-on access items. For Victor, "high-ticket offer design" is the lead bullet. Async chat, scheduled calls, ads consulting, etc. follow.

## Rule 7. Use the seller's credibility markers.

If the seller has paid for premium training, holds a relevant certification, has worked with a named client at a verifiable result, or has any other credibility marker, surface it where it builds trust on a specific deliverable.

For Timo helping Victor design a high-ticket offer: "I apply the framework I paid $15,000 to learn through the high-ticket coaching training I completed." That's credibility delivered exactly where the buyer needs it (right next to the offer-design deliverable).

**Why:** Specific credibility markers ($15K training, named clients, named programs) outperform generic claims (years of experience, "I've done this for X clients"). Buyers discount the generic; they trust the specific.

**How to apply:** During pre-draft confirmation, ask the user: "What credibility markers do you want surfaced, and where?" Place them next to the deliverable they support. Never invent a credibility marker; if the user can't name a specific source, the marker doesn't go in.

## Banned technical jargon in client-facing copy.

If you find yourself reaching for any of these in a client-facing proposal section, STOP and rewrite in plain English. These are internal/engineering language, not buyer language.

- "Voice-of-customer database" → "system that keeps track of what your audience is saying"
- "Persona extraction layer" → "personas pulled from what your audience is actually saying"
- "Quote-tagging schema" → "quotes from your audience tagged so we can pull them when we write"
- "Voice bank infrastructure" → "library of your phrasing that the system writes from"
- "Attribution dashboard" → "dashboard that shows where your leads come from"
- "VoC" → "your audience's voice"
- "ESP" (without first-mention expansion) → "email tool" or "email service provider (ConvertKit, ActiveCampaign, etc.)"
- "Schema," "layer," "infrastructure" as standalone words → describe what the thing DOES instead

This list grows. Add to it whenever a future draft slips technical jargon and gets corrected.

## Verdict gate before showing draft

Before pasting any draft to the user, run this self-check:

1. Does Section 1 lead with the engagement goal in plain language? If it leads with a list of components, FAIL.
2. Does each system block in Section 2 follow Rule 1's three-beat template (broken / walks away with / why)? If any block leads with "What Timo Builds," FAIL.
3. Is the consulting / ongoing access section a top-level section BEFORE pricing? If it's after pricing or buried inside pricing, FAIL.
4. Are there any market price anchors on the consulting itself ($X/month, $Y/hour comparisons)? If yes, FAIL.
5. Are systems ordered by client value (not build dependency)? If reordering would put a higher-impact system first, FAIL.
6. Are goal-driven deliverables (offer design, strategy work, program design) named as top-line items, or buried? If buried, FAIL.
7. Is technical jargon stripped (per the banned list)? If any survives, FAIL.

Only paste the draft to the user when all 7 pass.
