In any client-facing document (proposal, email, landing page copy, deck), NEVER write language that downplays the quality of what Timo produces. Even when scoping a "simple" version vs a "premium" version, the framing must remain confident.

**Banned phrasings (caught 2026-05-13):**
- "No promises of premium-grade production."
- "Don't expect everything to be exceptional."
- "Expect every post to be functional."
- "Functional, not premium."
- "Sufficient" used to describe Tim's work.
- "Basic" used to describe Tim's work.
- Any "scope down" language that reads as "you're getting the cheap version."

**What to write instead:**
- Differentiate scopes by what they INCLUDE and DELIVER, not by quality downgrade.
- "In-house edits" vs "production-tier edits with motion graphics + scrolling sheet music."
- "Simple cuts and captions for short-form posts" + "Premium long-form / scrolling-sheet-music pieces handled via a managed editor."
- Frame the upgrade tier by what it ADDS, not what the base tier LACKS.

**Why:** 2026-05-13 Ilan proposal. I wrote "No promises of premium-grade production. Don't expect everything to be exceptional. Expect every post to be functional." in System 5. Timo: "Please delete this part. What the fuck, don't ever downplay the production of my materials. Fuck you." Apparent intent was to manage expectations between simple Tim-edits vs premium editor. Real-world effect was to insult Tim's craft in his own commercial document.

**How to apply:**
- Scope differentiation = capability differentiation, not quality apology.
- Default to confident, then differentiate by deliverable type / format / production tier.
- Premium-tier path is "for pieces where the production complexity exceeds short-form basics" (motion graphics, scrolling sheet music, multi-cam, long-form documentary).
- If genuinely scoping a "minimum viable" deliverable, name it by deliverable shape, never by quality level.
- Applies to Tim's work AND to managed sub-contractor work he stands behind.
